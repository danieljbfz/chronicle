## Known bugs

Observed in the export output and the TUI. Each was checked against the code, and the notes below record where the problem actually lives — not where it first appeared. Most also want a regression test the existing suite was missing.

- [x] **A thinking block with no plaintext renders as a bare `> _Thinking_` quote.** Grounded against real history and the Anthropic docs. Claude stores two shapes of thinking part, both typed `"thinking"`. Most carry the full reasoning in the `thinking` key, and chronicle renders those correctly. A large share (on the author's machine, roughly 4,600 of 13,500) carry an empty `thinking` string and a long `signature`. That is the documented `display: "omitted"` mode. The full reasoning is encrypted inside the `signature` for multi-turn continuity, decryptable only by Anthropic's API and never written as readable plaintext on disk. The reasoning did happen, so this is not nothing. chronicle should keep a visible trace that the assistant thought here, rather than draw an empty quote or silently drop it. Render the encrypted case as an honest marker, for example `_[Thinking — recorded by Claude in encrypted form, no readable text]_`. The truly empty case is different and rarer (no text and no signature, so no reasoning occurred), and that one can be dropped. The two cases need to be told apart, which means `ThinkingBlock` has to carry whether a signature was present, since today it holds only `Text`. So the fix is not purely cosmetic. It touches the parser (read the signature, or at least a "was encrypted" flag), the `ThinkingBlock` contract (carry that flag), and the renderer (marker for encrypted, drop for truly empty). The behaviour is shared by the TUI and every export path through `steps.Markdown`. (Verified by exporting a session: parts with text render fine, signature-only parts produce the empty quote. See the extended-thinking docs at <https://platform.claude.com/docs/en/build-with-claude/extended-thinking>.)
- [x] **The three adapters disagree on empty thinking blocks, and none of them keep the encrypted trace.** Settled the shared rule: keep a thinking block with readable text, keep a flagged marker when the text is encrypted (Claude's signature-only shape), and drop one that is genuinely empty. Only the Claude adapter has the encrypted case (the copilot products write no signature), so the fix carries the rule in the Claude parser and the `ThinkingBlock.Encrypted` contract flag, while the copilot adapters' existing "keep text, drop empty" already matches the rule. Grounded against real data (5,949 of 12,851 thinking parts on the author's machine are the encrypted shape, and none are genuinely empty) and against Anthropic's extended-thinking docs. The Claude parser returns a `ThinkingBlock` even when the text is empty, so the empty quote above reaches the renderer. The copilot-chat parser adds a `ThinkingBlock` only when the text is non-empty, and copilot-agent guards every block the same way, so both silently drop an empty thinking block. Neither behaviour is fully right once the encrypted case matters. The shared rule should be: keep a thinking block when it has readable text, keep a marker when it is encrypted (Claude's signature case), and drop it only when it is genuinely empty. Settle that rule once and apply it across all three adapters so they behave alike.
- [x] **Copilot-chat sessions show as completely empty in the TUI.** Root cause found in the event-log replayer (`eventlog.go`), grounded against real VS Code data (34 of 72 multi-line sessions on the author's machine were affected). Two bugs compounded. First, `rawEvent.K` was typed `[]string`, but VS Code writes a patch path that descends through an array index as a JSON number, e.g. `["requests", 53, "response"]`; the number failed to decode, and because the streaming decoder cannot resync after a decode error the whole replay aborted, leaving only the snapshot before the first such patch — empty for a session whose content arrives through patches. Second, a kind-2 append value is the *array of elements* to append (VS Code sends a one-element array to append one item), not a single element; the replayer appended the whole array as one element. The fix types the path as `[]any`, walks it through both objects (string component) and arrays (numeric component) with a recursive write-back that grows arrays as needed, and extends the target array with each element of a kind-2 value. A previously-empty session now exports 5,359 lines of correct content end to end. The two original append-test fixtures were inaccurate (single-object append) and were corrected to the real array shape.
- [x] **Tool results render under a `User` heading, and the three adapters disagree on how to model them.** Fixed in the renderer: a turn whose blocks are all tool results is labeled `## Tool` regardless of the carrying record's role, so Claude's tool results (always the sole content of a user record — 31,400 of them on the author's machine, none mixed with other content) no longer read as something the person typed. The `(role, IsMeta)` convention is now consistent too: tool results are not meta in any adapter (the copilot-agent adapter no longer marks them meta), so `--no-tools` controls them everywhere and `--no-meta` leaves them alone. A turn that mixes tool results with prose keeps its role heading. Claude stores a tool result on a user-role record, so a pure tool-result turn renders as `## User`. The deeper problem is that each adapter models the same concept differently: Claude puts the `ToolResultBlock` on a `RoleUser` message (not meta), copilot-chat puts it on the `RoleAssistant` message (not meta), and copilot-agent puts it on a `RoleSystem` message with `IsMeta: true`. That inconsistency also changes filtering — `--no-meta` drops copilot-agent tool results but keeps the other two. Fix in the renderer by giving a `ToolResultBlock` its own heading regardless of the carrying message's role, and decide one consistent `(role, IsMeta)` convention for tool results across the adapters.
- [ ] **System and meta records render as bare turns.** A `No response requested.` line and `<local-command-stdout>` echoes are real user/assistant text records, so they surface as plain `## Assistant` / `## User` turns with raw markup. Decide which are bookkeeping that should be dropped or labelled, and render the rest without the raw tags.

  Grounded against real data on the author's machine, the patterns and their counts are: `<command-name>` slash-command invocations (447) — **already handled**, the parser's `isSlashCommandUserMessage` marks these meta; `<local-command-stdout>`/`<local-command-stderr>` command-output echoes on user/system records (388) — **not handled**, they render with raw tags; and an assistant record whose only text is exactly `No response requested.` (33) — **not handled**, pure bookkeeping. The recommended approach, which stays consistent with the existing slash-command handling rather than inventing a new mechanism: extend the meta-detection so a user record whose text is wrapped in `<local-command-stdout>`/`<local-command-stderr>` is marked meta the same way a slash command is (it is command machinery, not a prompt the person typed), and treat an assistant turn that is solely the `No response requested.` sentinel as meta too. This is a deliberate, heuristic call — it matches sentinel strings — so it wants its own focused pass rather than being folded into a larger change, and each pattern deserves a regression test built from the real shapes above.
- [ ] **The transcript screen has no per-session actions.** The transcript reader's footer offers only scroll and back (`transcript.go`) — there is no key to export, copy, or resume the open session. Add the bindings that make sense on a single conversation.
- [ ] **The TUI renders the conversation unfiltered, unlike every export path.** The transcript screen calls `steps.Markdown(conv)` directly, while `export`, `copy`, and `bulk_export` all run `steps.Filter` first. So the TUI cannot hide tools, thinking, or meta — the filter toggles the reading view was meant to have — and it shows messages the filtered exports drop. Add filter toggles to the transcript screen and make the two surfaces consistent.
- [x] **Only the Claude parser emits blockless messages.** A Claude user or assistant record whose content decodes to zero blocks still becomes a `Message` with an empty `Blocks` slice, and `writeMessage` renders that as a bare `## Role` heading with no body. Both copilot adapters already guard against this and skip a message once they see it has no blocks. `steps.Filter` drops these for the filtered export paths, but the unfiltered TUI shows the empty sections. The fix is to make the Claude parser skip a message with no blocks, the way the other two adapters do.
- [x] **Markdown rendering is fence-unsafe.** `writeBlock` dumps tool-result output, file-context content, and unknown raw JSON inside triple-backtick fences without escaping. Content that itself contains a ` ``` ` line closes the fence early and the rest renders as broken Markdown. Use a fence longer than the longest backtick run in the content, or escape it.
- [x] **Image and document tool results flatten to an empty string.** `flattenToolResultContent` in the Claude parser keeps only parts whose `type` is `"text"`. A tool result that returned an image (or any non-text part) flattens to `""`, so the `ToolResultBlock` renders as an empty fence with no sign of what came back. Surface a reference for non-text result parts the way `ImageBlock` and `DocumentBlock` do, rather than dropping them silently.
- [x] **Close the test gap behind these bugs.** The renderer now asserts on the rendered body for thinking blocks (the encrypted marker and the blockquote trailing-line trim), tool results (the `Tool` heading and the prose-keeps-its-role guard), and fence-breaking content. The system-record body tests land with that item above, since the fix and its tests belong together.
- [x] **The TUI dropped a background screen's messages when the user switched away mid-load (the stats loading freeze).** Found during the audit, grounded against the reported symptom ("the stats loading page got stuck after I switched screens; the time and animation froze"). The app router in `cmd/chronicle/tui/app.go` forwarded every non-intercepted message to the active screen only, while `WindowSizeMsg` alone was broadcast to all screens. A section loads lazily on first open and its load runs as a command, so switching away while it loaded delivered the in-flight spinner tick to the now-active screen (which rejected it by spinner id), stranding the loader's tick loop — a frozen glyph and clock — and, if the load finished while away, delivering the result to the wrong screen so the section stayed on its loading view forever. Fixed by routing keyboard and mouse input to the focused view and broadcasting every other (background) message to every screen, each ignoring the messages that are not its own. Regression tests pin both halves of the routing rule.
- [x] **A renderer performance test flaked under the parallel race suite.** Found during the audit. `TestView_StaysWellUnderTheFrameBudget` timed one `View` call against a hard 5 ms threshold, which has no margin against scheduler preemption when `go test -race ./...` runs every package at once, so it failed intermittently by fractions of a millisecond. Its docstring also claimed a 1 ms threshold while the code used 5 ms. Fixed by measuring the fastest of several renders (which reflects the render's own cost rather than the machine's load at one instant) against an 8 ms budget — half the 60-fps frame budget — which still fails the eleven-millisecond regression the test exists to catch.

## Codebase review findings

A reviewer read every package by layer and reported findings, and each one below was then validated against the code before landing here. The two high-severity items were confirmed by hand. The lower ones are real but narrower, and a couple are latent (no input triggers them today). The review also cleared a good deal of code as correct, which is recorded at the end so the next reader knows what was already checked.

### High

- [x] **A cross-device trash move can delete the source after a failed write, losing the file.** In `composition/fsmove.go`, `copyFile` runs `defer out.Close()` and then returns the `io.Copy` error, so the error from `Close` is discarded. A filesystem can accept every write and only report a failure (a full disk, a dropped network volume) when the file is closed. On the cross-device path, `moveFileOrDir` deletes the source as soon as `copyFile` returns no error, so a close-time failure leaves the source gone and the destination truncated. This is silent data loss inside the recoverable trash, which is the one place that must never lose data. Capture the `Close` error, and on any copy or close failure remove the partial destination and return the error before the source is deleted.
- [x] **`chronicle resume` can hang forever on a malformed session line.** In `adapters/claude/resume.go`, `readSessionCwd` drives a `json.Decoder` in a loop and continues past any non-EOF error, on the assumption (stated in the comment) that the decoder skips past a bad value on the next call. That assumption is false for a JSON syntax error. Go's `json.Decoder` stores a syntax error permanently and stops advancing, so every later `Decode` returns the same error and the loop spins forever. A truncated final line, which is plausible when a session file is read while Claude is still writing it, triggers this and the command never returns. Stop the loop on any non-EOF error and fall back to the folder-name decode, the same fallback the function already uses when a file carries no cwd.

### Medium

- [x] **The copilot-agent adapter stores raw JSON as a tool result, not the text.** In `adapters/copilotagent/parse.go`, the tool-result block is built with `Output: string(d.Result)`, where `d.Result` is the raw JSON of the result. The other two adapters flatten the result to its text before storing it, which is what the `ToolResultBlock` contract asks for. The reader sees a blob of JSON where the other providers show plain output. An existing test passes only because the expected words happen to appear inside the JSON. Flatten the result to text the way the copilot-chat adapter's `decodeToolResult` does.
- [x] **Thinking and away-summary blockquotes gain a spurious blank line.** In `steps/export.go`, the thinking and away-summary renderers split the text on newlines and prefix each line with `> `. When the text ends with a newline, which assistant content routinely does, the split produces a trailing empty piece and the renderer emits an extra `> ` line inside the blockquote. Trim the trailing newline before splitting. This sits in the same two `writeBlock` cases as the empty-thinking fix, so the two are worth doing together.
- [x] **The Claude orphan scan walks the projects tree twice.** In `adapters/claude/orphans.go`, `scanSecurityWarningOrphans` builds its own set of known session ids by walking every project directory, even though `PlanOrphanScan` already built that set and passed it to the companion scan. On a large install the tree is walked end to end twice per scan. Pass the already-built set in, the way `scanCompanionOrphans` already takes it.
- [x] **`TrashRestore` treats a stat error as "the destination is free."** In `composition/trash.go`, the pre-flight loop aborts the restore only when `os.Lstat` reports the destination already exists. Any other `Lstat` error, such as a permission denial on a parent directory, falls through and the restore proceeds, so the one safety check before a destructive move is only half-enforced. Treat a "not found" result as clear and return any other error.

### Low and latent

- [x] **A mutation through an array index would drop a copilot-chat conversation.** This was not latent — it was the cause of the empty-sessions bug above, and it bit on real data far more often than expected. Fixed there: `setAtPath`/`appendAtPath` now walk a numeric path component into a slice instead of overwriting it, and the path is typed `[]any` so a numeric index decodes in the first place. (The earlier framing assumed numeric components only ever arrived as string keys; in fact VS Code sends them as JSON numbers, which is what made the original `[]string` path abort the whole replay.)
- [ ] **A backslash in a project path could corrupt the sjson key path.** In `adapters/claude/global_config.go`, `sjsonPath` escapes dots and stars in a project key but not backslashes. A POSIX path never contains a backslash, so this does not bite today. A Windows path would, and the dangling-config edit could then delete the wrong key. Worth fixing only if chronicle starts targeting Windows.
- [ ] **`TrashRestore` does not reject `..` in the entry id.** In `composition/trash.go`, the entry id is joined onto the trash directory without rejecting path components, so a hand-typed id with `..` resolves outside the trash. Real damage needs a manifest to be present at the resolved path, so this is a footgun rather than a likely accident on a single-user machine. Reject an id that is not a bare path component.
- [ ] **The transcript screen's doc claims a key binding it does not have.** In `cmd/chronicle/tui/screens/transcript/transcript.go`, the package comment says the back action is bound to both Esc and `b`, but only Esc is wired. Either bind `b` or correct the comment.
- [x] **`steps.Filter` drops messages that arrive with no blocks, even with no flags set.** Resolved by the blockless-message fix above: every adapter now drops a blockless message at parse time, so none reaches `Filter`, and the only messages `Filter` drops are those it empties itself by hiding their last block. That drop is deliberate (a turn left with only a hidden tool call is noise) and the `Filter` docstring already documents it, so the behaviour is kept and no `steps` change is needed.

### Cleared by the review

The reviewers checked and confirmed these are correct, so the next reader does not need to re-examine them: the cobra command wiring and the `--apply` dry-run gating on every destructive command, the stdout-versus-stderr split, the rule that no entry point imports an adapter directly, the parallel detection in `composition/browse.go` (no data race, each goroutine owns its own pointer), the trash move-and-rollback path and its EXDEV gating, the TUI window-resize math and spinner-message forwarding, and the resilience canaries that keep unknown record and event types as `UnknownBlock`.

### Keep reviewing

The "Codebase review findings" section above was a first broad round across every layer, and it found real defects, which is a strong signal that more remain. A deliberate, exhaustive review of every package is still warranted and should continue — the recently reworked Claude parser, the event-log replay in the copilot adapters, the destructive paths in composition, and the TUI routing all deserve another careful read. Treat the lists above as what has been found so far, not the complete set.

## The TUI and web frontends

The terminal UI is the everyday interactive face of chronicle, built on the Charm v2 stack over the existing `composition.App`. The locked-in design rules (the `ui.Frame` contract, the section router and tab strip, the footer and Esc-ladder conventions, the canonical separators, the single-line row invariant) are documented in [`codebase-tour.md`](codebase-tour.md#the-terminal-ui). New screens follow them.

Built so far: the foundation, the session list, the transcript reader, and the stats view. Remaining:

- [ ] **Doctor screen.** Reads `composition.App.Doctor`. One card or table row per detected provider with the root, version, fingerprint, reachability, session count, and any warnings. Follow the stats screen as the structural template. The `cmd/chronicle/tui/screens/doctor/` package is a placeholder today.
- [ ] **Trash screen.** Reads `composition.TrashList`, with restore and empty actions through `composition.TrashRestore` and `composition.TrashEmpty`. Both actions are destructive and need a confirmation modal, which is a new shared `ui` component, not inline chrome.
- [ ] **Memory screen.** Reads `composition.ListMemories`. Project and global memories in one list with a scope filter, rendering a file on demand.
- [ ] **Web frontend.** A local, single-binary web app for browsing sessions and sharing rendered transcripts in a browser. Read-only in its first cut (browse, preview, search, export), with the destructive flows left to the TUI. The strategic framing is in [`feature-roadmap.md`](feature-roadmap.md#the-web-frontend).

## Performance

- [x] **Listing and stats fully parsed every session file to build a summary — this was the top performance problem.** Grounded against the author's data on 2026-06-01 (`chronicle stats` measured at 37.9 s real / 43.0 s user / 1.5 s sys, peak RSS 5.96 GB across ~2.2 GB of session files), the cost was confirmed CPU-bound on parsing, not I/O-bound on disk. Each adapter's `ListSessions` called `readSessionFile` on every session, building the entire `Conversation` — every record, every content block — only to keep five summary fields. Both `ListSessionsAll` and `Stats` walked this path, and so did `clean stale`. A shallow "read only the head" parse was rejected because the turn count is the length of the *folded* message list and Claude's model is the most-frequent per-message value, so both genuinely need every record, and copilot-chat's request count, title, and last-activity all come from the full event-log replay. Approximating them would have changed the reported numbers and grown a second, drift-prone parse path per adapter.

  **The fix, landed as a contract-and-composition change** (the architecture is documented in [`codebase-tour.md`](codebase-tour.md), under the `contracts/` and `composition/` walkthroughs): `Provider.ListSessions` was split into a parse-free `ListSessionRefs` (stat-only) and an expensive `SummarizeSession` (parse one). The triplicated summary-assembly collapsed into one `contracts.NewSessionSummary`. Composition (`session_list.go`) owns a persistent per-session cache (`summary_cache.go`) under `~/.cache/chronicle/summaries.json`, keyed on size, modification time, and the storage fingerprint, serving unchanged sessions for free and parsing only the misses in parallel. The cache is correct on append-only logs because any change moves the byte size, which is part of the key. Three callers that only needed identifiers — `clean abandoned`, `search`, `bulk_export` — moved to the cheap `ListSessionRefs`, removing a double parse they were paying. The adapters' disagreement on a corrupt session (Claude and copilot-chat failed the whole listing, copilot-agent skipped it) was unified on skip-and-continue.

  **Measured result:** `chronicle stats` went from 37.9 s to **0.10 s** on a warm cache and **18.8 s** cold (the parallel parse halved the cold path). The reported totals are unchanged — the 1.8 GB of copilot-chat data round-trips through the cache identically, and two consecutive runs are byte-identical.

- [ ] **Prune dead entries from the session-summary cache.** The cache in `composition/summary_cache.go` never removes an entry, so a deleted session leaves a dead entry behind forever. Each entry is a few hundred bytes, so the growth is harmless in practice (thousands of dead entries are a small file), which is why the first cut left it unbounded. The natural fix, when it becomes worth doing, is to drop any entry not seen during a full unfiltered listing pass — the same shape the freedesktop thumbnail cache uses with its age and size caps. Low priority until a real install shows a cache file large enough to matter.

## Claude transcript fidelity

- [ ] **`--no-rewound` prunes abandoned `/rewind` branches from the export.** When the user runs `/rewind` (or presses Esc twice), Claude restores the conversation to an earlier checkpoint and starts a new branch. The messages on the abandoned branch stay in the session file. chronicle sorts every record by timestamp today, so a rewound session shows the abandoned attempts interleaved with the surviving one, which reads as false starts.

  The flag drops the abandoned branches and keeps the surviving one. It stays **off by default**: chronicle is an archive, and a rewound attempt is real text the user typed, not machine bookkeeping. The name and polarity match the existing `--no-tools` and `--no-files`. The plan mirrors the `IsSidechain` / `HideSidechain` pair: add a `Rewound` field to `contracts.Message`, set it in the Claude parser, add `HideRewound` to `steps.FilterOptions`, and wire `--no-rewound` to it.

  The detection: a rewind branch-point is a record with two or more children that are genuine user prompts. At each such point, keep the child whose subtree contains the latest activity and drop the other subtrees whole. This signature never matches a parallel tool call (which branches into tool-result and assistant children, not prompt children) and never crosses a `/compact` boundary (which starts a new disconnected root).

  **Blocked on grounding.** There is no controlled, inspected sample of what `/rewind` and `/branch` (the `--fork-session` flag) actually write to disk. The way to ground it is to run `/rewind` once and `/branch` once in a throwaway session and read the resulting `.jsonl`. The grounding also settles the name: if `/branch` and `/fork` only ever write a separate session file, the only within-file abandoned branches come from `/rewind` and `--no-rewound` is precise. If a single file can also hold a fork-created branch, the broader `--no-abandoned` is the right name.

- [ ] **`--no-precompact` hides pre-compaction history.** `/compact` writes a new root (a `system` record, subtype `compact_boundary`, `parentUuid: null`) with a summary, and the pre-compaction conversation stays in the file as a separate tree. chronicle shows the full history today, which is the right archival default. Claude Code's resume view instead shows the summary plus the post-compaction messages. A `--no-precompact` flag (default off, matching the `--no-rewound` shape) would offer that resume-style view. Lower priority than the rewind prune, and a separate concern from it.
